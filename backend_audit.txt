────────────────────────────────

1. SYSTEM OVERVIEW
   ────────────────────────────────

* Identify the purpose of the application
* Identify expected user behavior and load
* Identify which operations are read-heavy vs write-heavy
* Identify which data must load immediately vs can be deferred

────────────────────────────────
2. DATABASE ARCHITECTURE
────────────────────────────────
For each table:

* Purpose of the table
* Estimated row count now and in 6–12 months
* Average row size
* Access pattern (read/write frequency)
* Required indexes
* Missing or unnecessary indexes
* Foreign key usage (and whether they should exist)
* Normalization vs denormalization tradeoffs
* Any table that should be split, merged, or archived

Also evaluate:

* Query patterns
* N+1 risks
* Full table scans
* Joins vs precomputed data
* Pagination strategy
* Whether materialized views should be used
* Whether caching is required (Redis, memory, edge)

────────────────────────────────
3. BACKEND ARCHITECTURE
────────────────────────────────

* Framework and runtime evaluation
* API structure (REST / RPC / GraphQL)
* Endpoint responsibilities
* Request lifecycle
* Serialization format (JSON vs alternatives)
* Data overfetching / underfetching
* Error handling design
* Logging and observability
* Rate limiting
* Background jobs and queues

────────────────────────────────
4. FRONTEND ↔ BACKEND DATA FLOW
────────────────────────────────

* What loads on initial app load
* What should NOT load initially
* Opportunities for lazy loading
* Streaming possibilities
* Partial responses
* Progressive hydration
* Server-side vs client-side fetching
* Bandwidth optimization

────────────────────────────────
5. HOOKS, STATE, AND DATA MANAGEMENT
────────────────────────────────

* Which hooks fetch data
* When they fire
* Whether they refetch unnecessarily
* Dependency array correctness
