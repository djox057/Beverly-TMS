1. ARCHITECTURE OVERVIEW
================================================================================

The /orders page uses a hybrid data loading strategy:
- Real-time database queries for UNLOCKED orders (with pagination)
- IndexedDB local cache for LOCKED (archived) orders  
- Supabase Storage for company-wide cache synchronization
- Real-time WebSocket subscriptions for live updates
- SERVER-SIDE SEARCH for finding ANY order in database

FILES INVOLVED:
---------------
- src/pages/Orders.tsx                    - Main UI component

2. DATA FLOW EXPLANATION
================================================================================

STEP 1: Initial Load
--------------------
1. Fetch first 100 UNLOCKED orders from database with all joins
2. Load LOCKED orders from archive cache
3. Merge and transform for UI display

STEP 2: Server-Side Search (NEW)
--------------------------------
When user types in search (2+ characters):
1. Debounce input (300ms)
2. Query DATABASE DIRECTLY via useOrdersSearch hook
3. Search across: load_number, broker_load_number, internal_load_number
4. Return up to 100 matching results from ALL orders (locked or unlocked)
5. Display search results instead of loaded orders

STEP 3: Load More Unlocked Orders (NEW)
---------------------------------------
When user clicks "Load More":
1. Use cursor-based pagination (created_at < last_order)
2. Fetch next 100 unlocked orders
3. Merge into existing cache
4. Update pagination state
